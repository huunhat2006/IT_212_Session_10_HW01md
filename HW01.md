# Bài 1: Phân Tích & Lựa Chọn Chiến Lược Tài Liệu Hóa

## Đáp án lựa chọn

**Chọn: Phương án B – Sử dụng Antigravity (version 1.23+) để index toàn bộ dự án và phân tích luồng nghiệp vụ.**

---

# Lý do lựa chọn phương án B

Phương án B là lựa chọn tối ưu nhất vì khai thác đúng thế mạnh của công cụ Antigravity trong Session 10: **phân tích toàn bộ codebase dựa trên ngữ cảnh đầy đủ (Full Project Context)** thay vì chỉ xử lý từng đoạn mã riêng lẻ.

### Ưu điểm

* **Có đầy đủ ngữ cảnh của toàn bộ dự án (Full Context)**

  * Antigravity index toàn bộ thư mục Guai-api.
  * AI có thể hiểu được mối liên kết giữa Controller, Service, Repository, Security Filter, Entity, DTO và các thành phần khác.
  * Không cần người dùng phải tự ghép các phần logic lại với nhau.

* **Theo dõi được luồng xử lý xuyên suốt (End-to-End Flow)**

  * AI có thể lần theo luồng:

    * API Endpoint
    * JWT Security Filter
    * Authentication
    * Controller
    * Service
    * Repository
    * Database
  * Nhờ đó tài liệu SRS phản ánh đúng quy trình nghiệp vụ thực tế.

* **Giảm thất thoát ngữ cảnh (Context Loss)**

  * AI không chỉ nhìn thấy một vài file riêng lẻ.
  * Có thể hiểu được các lời gọi hàm (method call), dependency injection, interface và class implementation.

* **Sinh tài liệu SRS có cấu trúc**

  * Prompt yêu cầu rõ:

    * Pre-conditions
    * Main Flow
    * Exceptions
  * AI có thể tổng hợp trực tiếp thành tài liệu nghiệp vụ thay vì chỉ giải thích code.

* **Tiết kiệm thời gian**

  * Không cần mở từng file.
  * Không cần copy code.
  * Không cần tổng hợp thủ công.

* **Phù hợp với các dự án lớn**

  * Các hệ thống Java Spring Boot thường có hàng trăm hoặc hàng nghìn file.
  * Antigravity được thiết kế để làm việc với codebase lớn nên hiệu quả hơn nhiều so với việc phân tích từng file.

# Phân tích hạn chế của phương án A

## Phương án A

*Mở từng file trong VSCode và hỏi Github Copilot Chat.*

### Ưu điểm

* Dễ thực hiện.
* Phù hợp khi cần hiểu nhanh một đoạn code nhỏ.
* Không cần index toàn bộ dự án.

### Hạn chế

* **Mất ngữ cảnh giữa các file**

  * Copilot chủ yếu nhìn vào file đang mở.
  * Không hiểu đầy đủ quan hệ giữa Controller, Service và Repository.

* **Người dùng phải tự tổng hợp**

  * AI chỉ giải thích từng đoạn code.
  * Việc ghép thành tài liệu SRS phải làm thủ công.

* **Khó theo dõi toàn bộ Checkout Flow**

  * Luồng nghiệp vụ bị chia thành nhiều phần.
  * Dễ bỏ sót bước xử lý hoặc điều kiện đặc biệt.

* **Không tối ưu với dự án lớn**

  * Phải mở rất nhiều file.
  * Mất nhiều thời gian.

### Rủi ro Context Loss

* AI không biết:

  * phương thức nào được gọi tiếp theo,
  * dữ liệu được truyền đi đâu,
  * các kiểm tra bảo mật nằm ở file nào,
  * repository nào thực sự truy cập dữ liệu.

Điều này có thể dẫn đến tài liệu SRS thiếu hoặc sai luồng nghiệp vụ.

# Phân tích hạn chế của phương án C

## Phương án C

*Copy thủ công 5 file Java lên ChatGPT/Gemini.*

### Ưu điểm

* Dễ sử dụng.
* Không cần cài đặt công cụ.
* Có thể nhận được tài liệu SRS tương đối nhanh nếu phạm vi nhỏ.

### Hạn chế

* **Giới hạn context**

  * Chỉ những file được copy mới được AI phân tích.
  * Nếu còn các class khác liên quan, AI sẽ không biết.

* **Dễ bỏ sót file**

  * Có thể quên:

    * DTO
    * Entity
    * Utility
    * Exception Handler
    * Config
    * Security
    * Mapper
  * Khi đó tài liệu sẽ không đầy đủ.

* **Không thể truy vết dependency**

  * AI không thể mở thêm các file được tham chiếu.
  * Nếu Service gọi sang một Service khác mà chưa được copy, AI sẽ không hiểu toàn bộ quy trình.

* **Không phù hợp với dự án lớn**

  * Số lượng file nhiều khiến việc copy rất mất thời gian.
  * Có thể vượt giới hạn độ dài đầu vào của mô hình AI.

### Rủi ro Context Loss

Đây là phương án có nguy cơ thất thoát ngữ cảnh cao nhất vì:

* AI chỉ nhìn thấy phần code được dán.
* Không biết còn những thành phần nào khác trong dự án.
* Có thể suy luận sai hoặc giả định thiếu chính xác về luồng xử lý.
